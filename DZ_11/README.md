Домашнее задание №11: Расчет ресурсов кластера Kubernetes

Цель работы
Научиться рассчитывать требования к ресурсам кластера Kubernetes под конкретный проект с учетом отказоустойчивости, системных резервов и запаса на выход из строя узлов.

Исходные данные

Кеш	4 ГБ RAM, 1 ядро	3	12 ГБ RAM, 3 ядра

1. Базовый расчет ресурсов
Суммарные требования приложения (нетто):

RAM: 12 + 12 + 0.25 + 6 = 30.25 ГБ

CPU: 3 + 3 + 1 + 10 = 17 ядер

Это минимальный объем ресурсов, необходимый для запуска всех подов без учета:

Системных резервов на узлах (kubelet, ОС, DaemonSets)

Запаса на отказ узла

Возможности масштабирования

2. Системные резервы на узлах
На каждом узле часть ресурсов резервируется для:

Операционной системы и системных служб

Kubelet

Eviction threshold (порог вытеснения)

DaemonSets (kube-proxy, сетевой плагин, логирование, NodeLocal DNSCache и др.)

2.1. Формулы расчета резервов
CPU резерв (согласно learnk8s.io):

6% первого ядра → 60m

1% второго ядра → 10m

0.5% третьего и четвертого ядер → 10m (на каждые два ядра)

0.25% на каждое последующее ядро

RAM резерв (согласно learnk8s.io):

255 МБ для машин с RAM < 1 ГБ

25% первых 4 ГБ

20% следующих 4 ГБ (4-8 ГБ)

10% следующих 8 ГБ (8-16 ГБ)

6% следующих 112 ГБ (16-128 ГБ)

2% памяти свыше 128 ГБ

Дополнительно 100 МБ eviction threshold

DaemonSets (оценка):

CPU: ~0.5 ядра на узел

RAM: ~0.5 ГБ на узел

2.2. Пример расчета для узла с 8 ядрами и 32 ГБ RAM
CPU:

Резерв: 60m + 10m + 10m + 10m = 90m ≈ 0.09 ядра

DaemonSets: 0.5 ядра

Доступно для пользовательских подов: 8 - 0.59 ≈ 7.4 ядра

RAM:

25% от 4 ГБ = 1 ГБ

20% от 4 ГБ = 0.8 ГБ

10% от 8 ГБ = 0.8 ГБ

6% от 16 ГБ = 0.96 ГБ

Eviction threshold: 0.1 ГБ

DaemonSets: 0.5 ГБ

Итого резерв: 1 + 0.8 + 0.8 + 0.96 + 0.1 + 0.5 = 4.16 ГБ

Доступно для пользовательских подов: 32 - 4.16 = 27.84 ГБ

Эффективность использования: ~87% RAM, ~92.5% CPU.

3.2. Анализ вариантов
Вариант A (4 узла по 32 ГБ, 8 ядер):

✅ Низкие накладные расходы (больше ресурсов доступно для подов)

✅ Быстрое масштабирование (добавление одного узла дает много ресурсов)

✅ Меньше сетевого трафика между узлами

❌ При отказе одного узла теряется ~25% мощности кластера

❌ Меньшая гранулярность масштабирования

Вариант B (8 узлов по 16 ГБ, 4 ядра):

✅ Более равномерное распределение подов

✅ При отказе узла теряется только 12.5% мощности

❌ Выше накладные расходы (больше системных резервов суммарно)

❌ Больше запросов к API (8×5 rps = 40 rps против 20 rps в варианте A)

Вариант C (15 узлов по 8 ГБ, 2 ядра):

❌ Очень высокие накладные расходы (~25% ресурсов теряется)

❌ Много узлов усложняет обслуживание

❌ Высокая нагрузка на API (75 rps)

✅ Максимальная отказоустойчивость (потеря одного узла незначительна)

3.3. Выбор оптимальной конфигурации
Для данного проекта выбираем Вариант A (4 узла по 32 ГБ RAM, 8 ядер) как наиболее сбалансированный по эффективности и стоимости. Он обеспечивает достаточный запас ресурсов и прост в обслуживании.

Итоговые параметры кластера:

4 worker-узла

Каждый узел: 8 vCPU, 32 ГБ RAM

Суммарная мощность: 128 ГБ RAM, 32 vCPU

Доступно для подов: ~111 ГБ RAM, ~29.6 vCPU (после резервов)

Запас относительно требований: ~3.7× по RAM, ~1.74× по CPU

4. Учет отказоустойчивости (N+1)
Необходимо гарантировать работоспособность при выходе из строя одного узла.

4.1. Распределение критических компонентов
Критические компоненты (база данных и кеш) должны быть распределены так, чтобы при потере одного узла не потерять больше одной копии каждого компонента.

Вариант размещения:

Узел	База данных (3 копии)	Кеш (3 копии)	Фронтенд (5 копий)	Бекенд (10 копий)
Node1	1 копия	1 копия	2 копии	3 копии
Node2	1 копия	1 копия	1 копия	3 копии
Node3	1 копия	1 копия	1 копия	2 копии
Node4	-	-	1 копия	2 копии
4.2. Проверка при отказе узла
Рассмотрим худший сценарий – отказ Node1 (самый нагруженный):

Потерянные ресурсы:

База данных: 1 копия (остается 2)

Кеш: 1 копия (остается 2)

Фронтенд: 2 копии (остается 3)

Бекенд: 3 копии (остается 7)

Оставшиеся ресурсы в кластере (3 узла):

Доступная RAM: ~83.5 ГБ

Доступный CPU: ~22.2 ядра

Требуется для оставшихся подов: ~20 ГБ RAM, ~11 ядер

Вывод: Кластер сохраняет работоспособность с запасом даже после отказа одного узла. Критические сервисы (БД и кеш) сохраняют кворум (2 из 3 копий).

4.3. Учет запаса на масштабирование
Оставшийся запас (~63 ГБ RAM, ~11 ядер) позволяет:

Увеличить количество копий бекенда на 50% (до 15)

Добавить дополнительные сервисы

Компенсировать возможный рост потребления ресурсов

5. Рекомендации по размещению подов
Для обеспечения отказоустойчивости рекомендуется:

Использовать PodAntiAffinity для критических компонентов (БД и кеш), чтобы Pod'ы не попадали на один узел.

Для бекенда и фронтенда использовать PodTopologySpreadConstraints для равномерного распределения по узлам.

Настроить PodDisruptionBudget для БД и кеша с minAvailable: 2.

Использовать Vertical Pod Autoscaler в режиме рекомендаций для точной настройки requests/limits.

Примечания
Расчеты приведены для Linux-узлов с учетом стандартных резервов kubelet и DaemonSets. Для Windows-узлов резервы могут отличаться.

В расчетах использованы requests подов. Если планируются limits выше requests, запас должен быть увеличен.

Для production-среды рекомендуется использовать NodeLocal DNSCache, что добавит ~0.1 ГБ RAM на узел.

Если используется Calico с eBPF, резервы CPU могут быть немного ниже.
